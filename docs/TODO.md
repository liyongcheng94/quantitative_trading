# 量化系统架构 TODO

> 未来实盘系统的四层架构蓝图 + 技术选型决策。当前 `qtrader` 是 Phase 1/2 学习用回测框架，本文档描述下一阶段（Phase 3+）要落地的实盘架构。

---

## 🏗️ 系统架构全景图（四层结构）

可以把这套系统想象成一个自动化的工厂流水线：

| 层 | 角色 | 职责 |
|---|---|---|
| **数据层**（Data Layer） | 原料仓库 | 囤积和清洗所有"原材料"（价量、财务、因子数据） |
| **决策层**（Decision Layer） | 大脑中枢 | 运行多因子模型，输出每日"买入/卖出/持有"信号 |
| **执行层**（Execution Layer） | 手脚 | 对接券商 API（如 miniQMT），将信号转化为实际委托单 |
| **监控层**（Monitoring Layer） | 仪表盘 | 盯系统健康度和账户风险，异常时向手机发送警报 |

---

## 1. 数据层（Data Layer）：杜绝"数据泄漏"的根基

代码质量要求最高的一层。目标不是存数据，而是存"对齐过的、永不改变的历史快照"。

### 存储方案

建议使用 **SQLite**（本地轻量级）或 **PostgreSQL**。使用 `date + stock_code` 作为联合主键。

### 核心数据表设计

- **`daily_quotes`**：存储 `open / high / low / close / volume / turnover_rate`（换手率）
- **`financial_indicators`**：存储 `pe / pb / roe / net_profit_yoy`（净利润同比）
  - 最关键字段：**`announcement_date`（实际公告日）**——防止未来函数的唯一保障

### 每日任务

每天收盘后（15:30）运行定时脚本：
- 增量更新当天数据
- 备份前一版本数据

---

## 2. 决策层（Decision Layer）：核心算法引擎

纯计算逻辑，不涉及任何 IO 操作，方便快速单元测试和回测。

### 模块拆分

| 模块 | 职责 |
|---|---|
| `factor_calculator.py` | 计算因子值（如 SUE、20 日动量、量价配合度） |
| `factor_processor.py` | 极值剔除（Winsorize 缩尾）、Z-score 标准化、中性化（对市值回归） |
| `model_predictor.py` | 加载训练好的 LightGBM 模型，输入处理好的因子值，输出综合得分（Score） |

### 信号生成逻辑

**只在每周五收盘后运行决策层**，输出下周的股票池。这是低频策略的纪律，避免被每日波动牵着鼻子走。

---

## 3. 执行层（Execution Layer）：连接虚拟与现实的桥梁

未来对接 miniQMT 的地方。核心原则：**"先模拟，再实盘"**。

### 实盘前置位（Simulation Mode）

连接实盘前，强制设计"模拟执行层"：程序产生信号后，只向本地日志写入"模拟委托记录"，并记录当时的买一/卖一价格。

### 滑点模型

实盘执行时，默认成交价是 `Signal_Price * (1 + 0.001)`（正向滑点千分之一）。**策略在扣除了这千分之一后依然盈利，才具备上实盘的资格。**

### 订单管理

维护一个 `Pending Orders` 队列：
- 委托单超过 **30 秒** 未成交 → 自动撤单
- 改用市价单（或买二价）重挂
- 确保在趋势行情中不会踏空

---

## 4. 监控层（Monitoring Layer）：数字"看门狗"

A 股低频交易不需要盯盘，但**必须盯系统**。本地搭建轻量级监控告警体系。

### 资产曲线监控

每天计算当前持仓的浮动盈亏。当日浮亏超过总权益的 **3%** → 强制触发"减半仓"指令（卖出 50% 持仓）。

### 物理心跳检测

每分钟运行一次脚本。检测到主程序卡死或电脑断网 → 调用飞书/钉钉机器人 Webhook 推送"⚠️ 量化系统离线"到手机。

### 日志归集

所有交易信号、下单记录、系统报错 → 统一通过 `logging` 模块写入文件，自动按周切分日志存档。

## 6. 技术选型：四个核心决策点

### 6.1 回测引擎：Backtrader vs 向量化自定义回测

| 方案 | 优点 | 缺点 |
|---|---|---|
| **Backtrader**（事件驱动型） | 功能完善、内置大量指标（Talib）、社区活跃、完美模拟实盘逐笔撮合、支持复杂条件单和分批止盈止损 | 慢（500 只股票 × 10 年数据要几分钟）、代码侵入性强（策略逻辑和框架深度耦合） |
| **向量化回测**（Pandas/NumPy） | 极快（几千只股票毫秒级）、适合大量因子迭代和参数搜索 | 无法完美模拟"卖不掉"、"涨停买不进"等实盘约束，容易过度乐观 |

**✅ 建议（低频场景）**：以**向量化回测为主力框架**。
- 低频 + 多因子选股，主要关注周度/月度调仓，对微观交易细节（滑点）敏感度低
- Pandas 回测能极大提高因子挖掘效率
- 仅在验证最终策略时，用 Backtrader 做一次"压力测试"

### 6.2 数据存储：数据库 vs 文件系统

| 方案 | 适用场景 |
|---|---|
| **SQLite/PostgreSQL**（关系型） | 结构化、按行更新的数据（日线行情、财务因子），查询灵活，方便 SQL 连接（如关联财报日期） |
| **Parquet/Feather**（列式文件） | 大规模、批量读取的数据。Pandas 原生格式，读取极快，磁盘占用小 |

**✅ 建议（A 股场景）**：采用 **"轻量混合架构"**。
- **SQLite**：只存 metadata（元数据）——股票列表、复权因子、财报日历。数据量小且经常查找
- **Parquet 分区**：日线数据按年份分区（如 `data/2024/stock_quotes.parquet`）。回测时只加载需要的年份进内存，利用 Pandas 向量化能力

### 6.3 实盘对接：miniQMT 的调用策略

提前为 miniQMT 预留接口。

**抽象层设计**：写一个 `BaseBroker` 类，定义 `buy()` / `sell()` / `query_position()` 等抽象方法。

**两层实现**：
- `LocalSimulateBroker`：模拟执行，只往本地日志写记录
- `MiniQMTBroker`：真正调用 `xtquant` 下单

**好处**：开发策略时完全不用关心底层是模拟还是实盘，只需把信号丢给 Broker。QMT 开通后，**切换一行配置代码即可**，无需改动策略逻辑。

### 6.4 调度器（Orchestrator）：工作流编排

低频策略不需要复杂分布式框架，但需要时间触发器。
- **Windows**：任务计划程序 触发 Python 脚本
- **Linux/Mac**：Crontab

**每日时间线设计**：

| 时间 | 任务 | 逻辑 |
|---|---|---|
| 15:45（收盘后） | 数据更新脚本启动 | 拉取当日日线数据，存入 Parquet |
| 16:00 | 因子计算脚本启动 | 计算当日因子值，存为临时缓存 |
| 17:00（周五） | 决策模型启动 | 加载模型 → 预测得分 → 生成"下周一"调仓清单 |
| 20:00 | 预埋单生成 | 根据调仓清单，计算下周一开盘委托价，写入待执行队列 |
| 次日 09:20 | 执行脚本启动 | 读取待执行队列，通过 Broker 发出委托 |

## 8. 实盘防御代码：五层"防御工事"

> 策略逻辑只占系统代码的 20%，剩下 80% 都是在处理"万一……怎么办？"。低频量化虽然交易频率低，但每一次决策都重仓，对系统稳定性要求反而更高。

### 8.1 数据断层熔断机制（Data Circuit Breaker）

策略模型依赖数据源（AkShare 或 miniQMT 行情）。数据源断流、延迟或返回空值时的正确反应：

- **异常检测**：每次拉取数据后，立即检查 DataFrame 是否为 None 或空；检查最新一条数据的 date 是否等于今天的交易日，延迟超过 5 分钟视为异常
- **熔断逻辑**：检测到数据异常 → 立即冻结所有新开仓信号，**保留对已有持仓的平仓能力**（平仓通常基于更简单的价格触发）
- **告警升级**：连续 3 次数据拉取失败 → 通过钉钉/飞书机器人发送 "🚨 数据源离线，策略已暂停" 的紧急通知

### 8.2 仓位错配与资金审计（Equity Reconciliation）

实盘中最容易被忽视的"沉默杀手"——本地数据库记录的持仓和券商账户实际持仓可能因撤单失败、网络延迟、复权突变而"对不上"。

- **开盘前强制对账**：每天早上 9:15 启动系统时，强制调用 `query_position()` 接口，核对本地持仓表与券商账户是否一致。若不一致 → 禁止任何调仓操作 + 发送告警 + 等待人工介入
- **资金预留**：计算可用资金时，手动减去 **1000 元** 作为"安全垫"，防止手续费/税费/价格四舍五入导致"资金不足"报错

### 8.3 网络延迟与订单状态机（Order State Machine）

下单后委托单经历 `已报 → 部成 → 全成 → 已撤` 等状态。低频策略通常用限价单，但限价单可能一天都无法成交。

- **超时撤单**：下单后启动 timer 计数器。订单在 **60 秒** 内未被全部成交，且市场价格已远离挂单价 → 主动撤单 + 推送提示
- **补单策略（重要）**：撤单后**不要立即用市价单追击**。低频策略建议改为"放弃本次交易，等待下一次周级信号"。急迫的市价单往往买在日内高点，长期严重侵蚀利润

### 8.4 物理服务器的"看门狗"进程（Watchdog）

不能指望策略脚本永远不死。Python 脚本可能因内存泄漏、第三方库报错而崩溃。

**外部守护进程**（不要在脚本内部处理所有异常，更可靠的方式是系统外运行 Shell 脚本无限循环检查）：

```bash
while true; do
    if ! pgrep -f "python main_strategy.py"; then
        echo "Strategy crashed! Restarting..." | mail -s "Alert" your_email@example.com
        python main_strategy.py &
    fi
    sleep 60
done
```

**股票池监控**：策略模型在"沪深 300"内选股，但某只股票因停牌被移除出指数，代码未及时更新 → 下单时接口报错。每周一强制重新拉取股票池列表，与本地缓存做差分比对，自动增删标的。

### 8.5 压力测试与"极端交易日"模拟（Stress Testing）

低频策略最容易暴毙的地方。回测可能完美，但实盘中政策大利好（千股涨停）或恐慌踩踏（千股跌停）会导致流动性瞬间枯竭。

- **流动性感知执行**：下单前计算"你的委托量 / 当前卖一档挂单量"。比例 **> 10%** → 强制拆单，将大单拆成每 5 分钟一笔的小单，缓慢吸筹，避免冲击成本
- **极端行情熔断**：上证指数单日涨跌幅超过 **4%** → 系统自动进入"只平仓、不开仓"防御模式。极端行情下量价规律暂时失效，模型胜率大幅下降

## 9. 工程化实践：基于 Git 的量化研究工作流

> 从"写完代码跑回测"到"构建一个可迭代、可复盘、可追溯的量化研究工厂"。借鉴成熟机器学习开发经验，彻底告别"Jupyter 文件名从 v1 改到 v29"的混乱状态。

### 9.1 数据与特征的版本控制（DVC + Git LFS）

量化研究的核心资产是数据和特征。不能用 Git 管理数据（太大），但必须管理"数据的版本"。

- **使用 DVC（Data Version Control）**：像 Git 管理代码一样管理数据和模型文件。原始数据的"哈希值"记录在 Git 仓库中，实际大文件存储在云端（阿里云 OSS）或本地共享盘
- **锁定数据源**：在 `dvc.lock` 文件中固定 AkShare / Tushare 的数据拉取日期。几个月后回头复现回测结果时，拉取的是同一份"历史快照"，避免"回测结果无法复现"的尴尬

### 9.2 实验追踪（Experiment Tracking）

调整因子权重或新增过滤条件后，需要知道对收益率的影响。**MLflow** 或 **Weights & Biases (W&B)** 完美胜任。

- **自动记录**：每次回测脚本运行结束后，自动调用 `mlflow.log_metrics()`，记录夏普比率、最大回撤、年化收益、调仓换手率等关键指标
- **参数绑定**：同时记录本次运行的模型参数（如动量窗口 20、止损阈值 0.1）
- **可视化对比**：在 MLflow UI 界面横向对比过去 50 次实验的收益曲线，快速定位哪些改动有效

### 9.3 回测结果校验：告别"幽灵收益"

将代码合并到 main 分支前，强制跑通"冒烟测试"。

- **前置校验**：在 CI/CD 流程（GitHub Actions）中加入检查代码——计算回测交易列表中的"买入日期"和"卖出日期"，确保没有任何一笔交易是在非交易日（如周末）发生。**这是极其简单但极其有效的"未来函数"检测器**
- **基准对比**：自动计算本次回测收益与同期沪深 300 收益的差值。跑输基准太多 → CI 流程直接标记"失败"，提醒策略逻辑可能出现了负 Alpha

### 9.4 单元测试：测试"原子逻辑"

量化代码同样需要单元测试。这是确保重构时不会改坏核心逻辑的唯一保障。

- **测试因子计算**：给定固定 open/high/low/close 数据，手动计算预期 MA5 值，与 `calculate_ma()` 函数输出对比
- **测试撮合逻辑**：模拟"涨停买不进"场景，验证回测引擎能正确跳过这笔交易，而不是以涨停价买入
